# Отчёт по лабораторной работе №2

## Цель работы

Нужно было реализовать ETL-пайплайн для лабораторной по дисциплине «Анализ больших данных»:

1. загрузить 10 CSV-файлов в PostgreSQL;
2. построить в PostgreSQL модель данных «звезда»;
3. с помощью Spark сформировать 6 аналитических витрин в ClickHouse;
4. завернуть всё в Docker Compose;
5. подготовить инструкции по запуску и проверке.

## Что было сделано

В репозитории собрана полностью воспроизводимая среда:

- `PostgreSQL 15` хранит raw-слой и схему «звезда»;
- `Apache Spark 4.0.2` выполняет обе ETL-джобы;
- `ClickHouse 24.3` хранит аналитические витрины;
- `docker-compose.yml` поднимает все сервисы одной командой;
- bash-скрипты автоматизируют подготовку CSV, скачивание JDBC-драйверов и запуск пайплайна.

## Анализ исходных данных

Перед реализацией были просмотрены исходные CSV-файлы и проверена кардинальность основных сущностей.

Наблюдения:

- всего файлов: `10`;
- строк в каждом файле: `1000`;
- всего строк после загрузки: `10000`;
- количество колонок: `50`;
- формат дат в raw-данных: `M/d/yyyy`;
- поле `product_description` может содержать многострочный текст.

Кардинальность некоторых сущностей:

- `customer_email`: `10000` уникальных значений;
- `seller_email`: `10000` уникальных значений;
- `store_name`: `383` уникальных значений;
- `store_name + store_city + store_country`: `10000`;
- `supplier_name + supplier_country`: `6295`;
- `product_name`: `3`;
- `product_name + product_category + product_brand`: `3233`;
- `sale_date`: `364` уникальные даты.

Из этого следуют два важных вывода:

1. часть сущностей в мок-данных почти полностью уникальна на уровне строки, поэтому готовых стабильных ключей для размерностей нет;
2. продукт в этих данных логичнее трактовать как SKU-уровень, а не только как `product_name`, потому что у одинаковых названий отличаются другие атрибуты.

## Архитектура решения

Пайплайн реализован в два этапа.

### Этап 1. CSV -> PostgreSQL raw -> schema star

1. Скрипт `scripts/copy-csv-files.sh` копирует исходные файлы в `postgres/data/csv/` и переименовывает их в `mock_data_0.csv ... mock_data_9.csv`.
2. PostgreSQL при первом старте выполняет SQL-скрипты:
   - `01-create-raw-table.sql`
   - `02-create-star-schema.sql`
   - `03-load-csv.sql`
3. Таблица `mock_data` наполняется `10000` строками.
4. Spark-джоба `spark/etl_to_star.py`:
   - читает `mock_data` по JDBC;
   - обрезает пробелы в строковых колонках;
   - парсит даты;
   - строит измерения `dim_customer`, `dim_seller`, `dim_product`, `dim_store`, `dim_supplier`, `dim_date`;
   - строит таблицу фактов `fact_sales`.

### Этап 2. PostgreSQL schema star -> ClickHouse marts

Spark-джоба `spark/etl_to_clickhouse.py` читает таблицы схемы «звезда» и записывает в ClickHouse 6 витрин:

- `mart_sales_by_product`
- `mart_sales_by_customer`
- `mart_sales_by_time`
- `mart_sales_by_store`
- `mart_sales_by_supplier`
- `mart_product_quality`

## Ключевые технические решения

### 1. Суррогатные ключи через hash + row_number

Для размерностей использован следующий подход:

- формируется естественный ключ как `sha2(concat_ws(...), 256)`;
- после удаления дублей для каждой размерности задаётся surrogate key через `row_number()`.

Это позволило избежать проблем с исходными ID из mock-данных, которые не годятся как надёжные ключи для объединения между файлами.

### 2. Даты приводятся в Spark, а не в PostgreSQL

В raw-таблице даты хранятся строками. Причина в том, что исходный формат `M/d/yyyy` удобнее и безопаснее привести в Spark, где это делается явно через `to_date(...)`.

### 3. Spark запущен в контейнере в режиме `local[*]`

Изначально рассматривалась схема master/worker, но для лабораторного датасета она не даёт практического выигрыша.

Финальное решение:

- использовать официальный образ `apache/spark`;
- держать контейнер Spark постоянно поднятым;
- запускать джобы через `docker compose exec`;
- выполнять Spark в режиме `local[*]`.

Для `10000` строк это проще, стабильнее и легче проверяется преподавателем.

### 4. Витрины ClickHouse строятся поверх уже подготовленной схемы «звезда»

Это разделяет ответственность между джобами:

- первая отвечает за очистку и моделирование данных;
- вторая отвечает только за аналитические агрегаты и запись в ClickHouse.

## Основные сложности и как они были решены

### 1. Имена CSV-файлов с пробелами и скобками

Исходные файлы назывались вида `MOCK_DATA (1).csv`. Для `COPY` в PostgreSQL это неудобно, поэтому был добавлен отдельный этап копирования и переименования.

### 2. Конфликт хостовых портов

Чтобы не зависеть от локально установленного PostgreSQL и других сервисов, были выбраны отдельные порты на хосте:

- PostgreSQL: `15432`
- ClickHouse HTTP: `18123`
- ClickHouse Native: `19000`

Внутри Docker-сети сервисы продолжают общаться по стандартным внутренним портам.

### 3. Недоступность первоначально задуманного Spark-образа

В процессе настройки выяснилось, что вариант с другим образом Spark оказался менее надёжным для воспроизводимого запуска. В результате решение было упрощено до официального контейнера `apache/spark:4.0.2-scala2.13-java17-python3-r-ubuntu`.

### 4. Очистка таблиц перед повторным запуском

Попытка выполнять `TRUNCATE` через JDBC внутри Spark дала лишнюю сложность. В финальной версии очистка выполняется обычными CLI-инструментами:

- `psql` для PostgreSQL;
- `clickhouse-client` для ClickHouse.

Так скрипты стали проще и предсказуемее.

### 5. Аутентификация в ClickHouse

Для JDBC-подключения был создан отдельный пользователь `spark` с правами на базу `bigdata`. Это убрало проблемы с подключением через `default`-пользователя.

Важно: большие размеры части витрин объясняются природой мок-данных. На полном наборе атрибутов многие сущности фактически уникальны на уровне строки, поэтому в витринах сохраняется мелкое зерно данных. Для учебной работы это допустимо, а в реальном проекте поверх этих же данных можно было бы добавить ещё один слой бизнес-агрегаций.

### Примеры проверенных результатов

По витрине времени:

- в `mart_sales_by_time` получилось `12` строк, по одной на каждый месяц `2021` года;
- максимальная выручка по месяцам получилась в `October 2021`: `228743.32`.

По витрине качества продукции:

- глобальная корреляция между рейтингом товара и количеством проданных единиц получилась около `0.001005`;
- это означает, что в данном mock-наборе заметной линейной зависимости практически нет.

По витрине продаж по продуктам:

- в топе по выручке встречаются повторяющиеся `product_name`, потому что витрина хранит продукт на уровне строки измерения `dim_product`, то есть фактически на уровне SKU.

## Автоматизация запуска

Для проверки лабораторной достаточно выполнить одну команду:

```bash
./scripts/run-all.sh
```

Этот скрипт:

1. готовит CSV;
2. скачивает JDBC jars;
3. пересоздаёт docker volumes;
4. запускает сервисы;
5. ждёт загрузки исходных данных;
6. запускает обе Spark-джобы;
7. выводит контрольные SQL-запросы.
